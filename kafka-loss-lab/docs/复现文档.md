## 问题一：Producer / CLI — 异步发送未确认（`acks=0`）

### 现象摘要

| 项目 | 说明 |
| --- | --- |
| 角色 | Producer / CLI |
| 配置要点 | `acks=0`：不等待 broker 确认即视为发送完成 |
| 典型触发 | 网络抖动、Broker 挂掉、Leader 切换 |
| 可补救性 | 不可补救（消息可能从未到达 broker） |
| 风险 | 客户端侧可能仍统计为「成功」，但 topic 中实际条数不足 |

### 原理说明

`acks=0` 时生产者不等待 broker 的确认；请求若在途或 leader 故障，数据可能未持久化，但应用层已继续发送下一批。此类丢失无法在事后通过 Kafka 重试「补写」到同一语义（消息可能从未入队）。

### 前置条件

- Docker（用于启动仓库内 `docker-compose` 集群）
- JDK 17+、Maven 3.9+（构建 CLI）

### 复现步骤

#### 1. 启动集群并创建 topic

在仓库根目录执行：

```bash
cd kafka-loss-lab
./scripts/up.sh
./scripts/create-topic.sh loss-lab 3 3 2
```

说明：多副本 + `min.insync.replicas=2`，便于在杀 broker 时观察 leader/ISR 变化。

（等价创建命令，可选：）

```bash
java -jar target/kafka-loss-lab-1.0.0.jar create-topic \
  --topic loss-lab \
  --partitions 3 \
  --replication-factor 3 \
  --min-isr 2
```

需先执行 `mvn -q -DskipTests package` 生成 JAR。

#### 2. 构建 CLI

```bash
mvn -q -DskipTests package
```

产物：`target/kafka-loss-lab-1.0.0.jar`。

#### 3. 风险生产（核心参数）

```bash
java -jar ./target/kafka-loss-lab-1.0.0.jar produce \
  --topic loss-lab \
  --count 100 \
  --acks 0 \
  --enable-idempotence false \
  --sync false \
  --linger-ms 200 \
  --sleep-ms 0
```

- `--acks 0`：`ProducerConfig.ACKS_CONFIG = 0`
- `--sync false`：异步 `send` + 回调
- `--enable-idempotence false`：与 `acks=0` 搭配（幂等通常要求更强 ack）

#### 4. 发送过程中制造故障

**另开终端**，在步骤 3 仍在运行时执行（示例杀掉 broker 1，可按实验改为 1～5）：

```bash
./scripts/kill-broker.sh 1
```

用于模拟 leader 不可用或网络路径中断；若需针对「当前分区的 leader」做精确操作，可先查该 topic 分区 leader 再选择要停止的 broker。

#### 5. 消费并核对条数

```bash
java -jar target/kafka-loss-lab-1.0.0.jar consume \
  --topic loss-lab \
  --group loss-lab-safe \
  --max-messages 20000
```

对比：生产端输出的成功条数（或日志中的 ack 统计）与消费到的消息条数。

### 常见问题：长时间无输出、像超时、几乎没发出去

1. **集群未就绪或未监听本机端口**  
   默认 `--bootstrap` 为 `localhost:19092,…,59092`（见 `CliConstants.DEFAULT_BOOTSTRAP`）。请先执行 `./scripts/up.sh`，并用 `docker ps` 确认 5 个 broker 已起来；必要时用仓库内 `scripts/kafka-docker.sh` 或容器内 `kafka-topics.sh --bootstrap-server localhost:9092 --list` 验证。

2. **Topic 不存在**  
   若集群关闭自动建 topic，需先执行步骤 1 中的 `create-topic.sh` 或 `create-topic` 子命令，否则生产端可能在拉元数据或发送阶段阻塞/失败。

3. **异步大批量 + 旧版固定 30s 等待过短**  
   `produce` 在 `--sync false` 时会等齐所有异步回调再汇总；若条数多（如 10000）且带 `--sleep-ms`，原先若固定只等 30 秒，会在回调未全部完成时退出，表现为统计不完整或像「超时」。当前实现已改为按条数与 `--sleep-ms` 计算更长的等待时间；若仍见 `警告: … 秒内未等齐全部异步回调`，可暂时减小 `--count` / `--sleep-ms` 做连通性验证。

4. **Broker 全挂或网络不通**  
   发送可能在本地缓冲区塞满后阻塞，直至 `delivery.timeout.ms`（见 `KafkaClientProps`）等超时，表现为长时间卡住；需恢复集群或检查防火墙/端口。

### 预期结果

- Topic 侧配置为 `replication.factor=3`、`min.insync.replicas=2`，生产端为 `acks=0`、`enable.idempotence=false`、异步发送。
- 发送过程中 kill 承载 leader 的 broker 后：**生产端仍可能出现大量「成功」类输出，但实际消费到的消息数小于发送成功数**，即「表面成功、实际未落盘」的不可补救丢失。

### 清理环境

```bash
./scripts/down.sh
```
