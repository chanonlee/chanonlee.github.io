网上很多数据结构练习无法拿来即用套用在工程代码里，原因是在实际设计中还需要考虑很多可能，如程序中断、内存不足、业务条件限制。最近在编程中遇到了一些场景，感觉抽象出来很有意思，列了几种可能方案看一看。

### 问题背景

一家大型连锁餐饮公司在多个地区开设了众多门店，这些门店分布在四级行政区划之下。现需设计一种配置方案，使得用户能够便捷地选择某一区域内的所有门店或单独选择特定门店进行配置绑定。在此过程中，必须确保每个门店只能被绑定一次；若尝试重复绑定，则系统应反馈错误信息，例如：“[xx区域/xx门店]已被绑定至[yy节点]”。此外，展示配置结果时，应按照所选的最大区域范围来呈现。

### 问题抽象

1. 存在一个五层的树状结构，每个节点可以由任何个体占有。
2. 当某节点被占据时，其下的所有子节点也将自动归属于同一占有者。绑定节点时，若该节点有任意子节点被占有，该节点无法被直接绑定。
3. 若该节点所有子节点都被同一个体占有，则该节点自动被该个体占有。
4. 如果试图占有某节点而其下存在已经被其他个体占有的子节点，则此次操作无效，并需返回一个标识了最大被占有子节点的信息。
5. 尽管树的整体深度固定不变，但节点之间的相对位置可能会随时间调整。

需要提供的基本接口：

1. 新增节点
2. 变更节点
3. 占有节点
4. 取消占有节点
5. 查询当前节点最大被占用的子节点
6. 查询该占有者占有的全部节点

### 数据库设计方案

#### 方案A 重写轻读

此方案建议使用单一表格来存储整个树结构，通过记录每个节点及其直接上级节点的方式来表达层级关系。对于规模较小的树（即节点数量有限），可以通过一次性加载所有相关数据到内存中来进行高效的查询与更新操作。

```java
public void updateAncestorsStatus(Node root, Map<String, Node> nodesByCode){
    if(root == null){
        return;
    }
    List<Node> childNodes = findSubs(root);
    if(CollectionUtils.isNotEmpty(childNodes)){
        Boolean hasSameOccupier = true;
        int unoccupiedCount = 0;
        String occupierCode = childNodes.get(0).getOccupierCode(); 
        for(Node node : childNodes){
            if(occupierCode == null){
                unoccupiedCount++;
            }else if(!occupierCode.equals(node.getOccupierCode()){
                hasSameOccupier = false;
            } 
        }
        if(unoccupiedCount == 0){
            // 如果没有任何被占用节点，则处于未占用状态
            root.setStatus = 0;
        }else if(unoccupiedCount == childNodes.size() && hasSameOccupier){
            // 如果所有子节点被同一个人占用，则处于全占用状态
            root.setStatus = 2;
            root.setOccupierCode = occupierCode;
        }else{
            // 如果都不是，则处于半占用状态
            root.setStatus = 1;
        }
    }
    return updateAncestorsStatus(nodesMap.get(root.getParentCode()), nodesByCode);
}

public void updateSubLevel(Node root, Map<String, Node> nodesByCode){
    Node parent = nodesByCode.get(root.getParentCode());

    // 若父级别跟当前节点-1相同，则返回
    if(parent.getLevel().equals(root.getLevel()-1)){
        return;
    }else{
        // 根据父类级别，调整所有子级别
        root.setLevel(parent.getLevel()-1);
    }
    List<Node> subNodes = findSubs(root);
    if(CollectionUtils.isNotEmpty(subNodes)){
        for(Node node : subNodes){
            node.setLevel(root.getLevel()-1);
            updateSubLevel(node);
        }
    }
}
```

```java
private List<Node> findOccupiedChildren(Node node, Map<String, Node> nodesByCode) {
    List<Node> occupiedChildren = new ArrayList<>();
    Queue<Node> queue = new LinkedList<>();
    queue.offer(node);
    
    while (!queue.isEmpty()) {
        Node current = queue.poll();
        for (Node child : getChildren(current, nodesByCode)) {
            if (child.getStatus() == 2) {
                occupiedChildren.add(child);
            } else if (child.getStatus() == 1) {
                queue.offer(child);
            }
        }
    }
    return occupiedChildren;
```

#### 方案B 拓展性强

考虑到未来可能存在的扩展需求，本方案提出将树形结构本身与具体的绑定逻辑分离，分别存储于不同的表格中。这样做的好处在于提高了系统的灵活性和可维护性。

当处理的数据量不大时，用采取全量加载的方式以简化开发流程。绑定只存在于叶子节点上，每次更新时需要把整棵树取出来渲染。

本表只是一个基础的数据关系表，实际使用中会为了查询做一些冗余字段，如保存所有节点路径，使用Like来寻找子节点。这会增加更新难度。

#### 方案C 有向无环图

表设计拆分为节点和关系，关系可以维护为有向无环图。但需要在代码中小心检查，以避免写入环。代码层面不一定靠谱，可能需要联合数据库的存储过程来写。

```sql
CREATE TABLE nodes (
    id INT PRIMARY KEY AUTO_INCREMENT,        -- 节点ID
    name VARCHAR(255),                        -- 节点名称
    type ENUM('region', 'store'),             -- 节点类型（region: 区域, store: 门店）
    level INT                                 -- 节点的层级（例如，1=省，2=市，3=区，4=门店等）
);

CREATE TABLE node_relations (
    id INT PRIMARY KEY AUTO_INCREMENT,        -- 关系ID
    from_node_id INT,                         -- 起始节点ID（父节点）
    to_node_id INT,                           -- 目标节点ID（子节点）
    FOREIGN KEY (from_node_id) REFERENCES nodes(id) ON DELETE CASCADE,  -- 外键，起始节点
    FOREIGN KEY (to_node_id) REFERENCES nodes(id) ON DELETE CASCADE     -- 外键，目标节点
);

CREATE TABLE bindings (
  id INT PRIMARY KEY AUTO_INCREMENT,
  node_id INT, -- 被绑定的节点ID
  binder_id INT, -- 绑定者（用户ID）
  unique KEY (node_id) -- node_id为唯一键
);
```


通用查询：

```sql
-- 查询某个节点的所有父节点
SELECT n.*
FROM nodes n
JOIN node_relations nr ON nr.to_node_id = n.id
WHERE nr.from_node_id = ?;

-- 查询某个节点的所有子节点
SELECT n.*
FROM nodes n
JOIN node_relations nr ON nr.from_node_id = n.id
WHERE nr.to_node_id = ?;

-- 查询某个节点所有后代
WITH RECURSIVE descendants AS (
    SELECT id, name FROM nodes WHERE id = ?  -- 起始节点
    UNION
    SELECT n.id, n.name
    FROM nodes n
    JOIN node_relations nr ON nr.from_node_id = n.id
    JOIN descendants d ON nr.to_node_id = d.id
)
SELECT * FROM descendants;

-- 查询某个节点所有祖先
WITH RECURSIVE ancestors AS (
    SELECT id, name FROM nodes WHERE id = ?  -- 起始节点
    UNION
    SELECT n.id, n.name
    FROM nodes n
    JOIN node_relations nr ON nr.to_node_id = n.id
    JOIN ancestors a ON nr.from_node_id = a.id
)
SELECT * FROM ancestors;
```

### 总结

综合比较三个方法，方案A是针对特定情境特写的，代码最复杂。但在目前的问题抽象下其实有点没必要，除非需要做数据库高频读。但如果高频都低频写，可以直接使用缓存。方案B的表结构拆分是最合理的，将业务属性和图结构拆分开，使用数据库乐观锁，是更好的实现。方案B是方案C的特定场景，省一个表，可以看需求取用。





### 拓展问题1：在结点数量庞大且服务内存资源有限的情况下，如何设计系统以确保一致性和原子性？

当面临大规模节点数量且内存资源受限的问题时，可以考虑采取分批处理策略来缓解内存压力。具体而言，应引入一种批处理机制，其中每个处理步骤都需被详细记录，以避免一次性加载所有节点导致的内存溢出风险。为了确保系统的最终一致性，可以采用重试机制来处理可能出现的部分失败情况。然而，在这种情况下，原子性难以完全保障，因此建议从应用层面设定合理的写操作窗口期，并将其与批处理过程分离，从而减少潜在的数据不一致风险。

### 拓展问题2：基于数据集大小，何时适合采用基于内存的解决方案，又何时转向非内存存储方案更为合适？

选择内存方案还是非内存存储方案主要取决于两个关键因素：

1. 整个树结构所有节点加载到内存中所需的容量，这可以通过预估数据总量来进行大致计算；
2. 批量读取及批量写入操作对底层数据库造成的性能影响。

通常来说，即使是对拥有上百万条记录的表格执行基本查询操作，其对数据库的负载也相对较小；但一旦这些数据需要被加载进内存，则必须谨慎评估由此带来的内存消耗问题。数据库瓶颈晚于内存瓶颈。
