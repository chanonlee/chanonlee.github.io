## 1. 问题: 如何让大模型稳定输出

我们有时候会遇到，问大模型同样的问题，每次回答都不一样。这在工程场景中是一个很大的问题：难以调试、难以复现、难以稳定集成。即使清除了上下文、使用了同样的模型、使用完全相同的prompt，回答也次次不同，该怎么解决这个问题呢？

方法其实很简单，设置参数即可。

```text
temperature = 0
```

## 2. 原理一：模型逐token生成

这里有个很反直觉的事。

LLM 并不是“输入一个 prompt，然后一次性输出一整段话”。它的生成过程，其实是**逐 token 进行的**，模型根据自己说过的话继续往下说，就像下面这样

```text
输入：prompt
输出：token1

输入：prompt + token1
输出：token2

输入：prompt + token1 + token2
输出：token3
...
```

那么，每一步是如何决定输出哪个 token 的？

答案是：**概率分布**。

在每一步生成时，模型都会根据当前上下文，给所有候选 token 分配一个概率，然后从中选择下一个输出。

## 3. **原理二：temperature 控制随机性**

我们假设模型有token候选词A~F，概率分别如下：

```text
Token候选：
A: 0.40
B: 0.25
C: 0.15
D: 0.10
E: 0.06
F: 0.04
```

那么问题来了：模型在这一刻，应该选择哪个 token？是始终选择概率最高的 A？还是在这些候选中“随机挑一个”？

这个选择方式，正是由一个关键参数**temperature**控制的。

当设置temperature后，模型的选择概率是这样的

```text
temperature = 0        → 只选 A（完全确定）
temperature = 0.7      → A概率高，但B/C也有机会
temperature = 1        → 原始分布
temperature = 1.5      → 更平均，更随机
```

可以这样理解：

- temperature 越低，模型越“稳定”；
- temperature 越高，模型越“发散”。

## 4. 原理三：token筛选

除了通过 temperature 控制随机性，在实际应用中，还可以通过限制候选范围来约束模型输出，避免采样到过于冷门或不合理的词。这类方法称为 **token 筛选策略**，主要包括 top_k 和 top_p。

### **top_k**

仅保留概率最高的 K 个 token，其余全部丢弃，再在这 K 个候选中进行采样。

> 👉 只在“前 K 名”中选择

### **top_p**

按概率从高到低排序并累加，直到累计概率达到阈值 p，仅保留这一部分 token。

> 👉 只保留“覆盖主要概率质量”的候选集合

### **三者协同机制**

```text
原始概率分布
(A:0.40 B:0.25 C:0.15 D:0.10 E:0.06 F:0.04)
        │
        ▼
【Step 1：Top-k / Top-p 过滤】
        │
        ├── top_k=3  → 只保留 A B C
        │
        └── top_p=0.8 → 保留 A B C（累计到80%）
        │
        ▼
候选集合（A B C）
        │
        ▼
【Step 2：Temperature 调整分布】
        │
        ├── T = 0     → A:100%
        ├── T = 0.7   → A > B > C
        ├── T = 1     → 原始比例
        └── T = 1.5   → A ≈ B ≈ C
        │
        ▼
【Step 3：采样输出】
        │
        ├── 可能输出 A
        ├── 或 B
        └── 或 C
```

一句话总结：**top_p / top_k 决定“选谁”，temperature 决定“怎么选”。**

## 6. cursor等编程Agent里可以设置temperature参数吗？

**不能直接设置。**

在工程场景中，如果需要进行编程、数学计算等对结果一致性要求较高的任务，通常会将 temperature 设置为 0，以保证模型输出的稳定性和可复现性；而在生成图片描述、文本创作等更偏创意的任务中，则会适当提高 temperature，以获得更丰富、多样的结果。

大多数编程类 Agent（如 Cursor）为了避免代码生成过程中的不稳定性，通常会在内部预设合适的参数组合，并不对用户开放这些配置。可以理解为： **这些工具已经在底层帮你调好了参数。**

## 7. temperature = 0带来的问题——死循环

稳定并不总是好事。

当 temperature = 0 时，模型每一步都会选择“当前最优解”，再将已输出的结果作为输入，生成下一个词。当重复一定次数，上下文稳定不再变化，模型就会持续输出重复词语，陷入死循环。

我把复现这个问题的方式写在这里。

我使用 LM Studio 运行了一个本地小模型（qwen3-vl-4b），并通过 LangChain 调用。

**LM Studio设置**

{% asset_img lm-studio-settings.png %}

其他参数设为默认。此时程序里的temperature不为0，不过问题不大，传入参数会强制覆盖。

**调用代码**
```python
from langchain_openai import ChatOpenAI

# =====================模型层=====================
# 此处可设置为本地模型或远程模型

llm = ChatOpenAI(
    base_url="http://127.0.0.1:1234/v1",
    api_key="lm-studio",
    temperature=0,
    model="local",
)

# =====================提示层=====================

prompt_text = "Write a sad poem about ducks, gooses."

# =====================调用层=====================

response = llm.invoke(prompt_text)

# =====================输出层=====================

print(response.content)
```

假设某一时刻，模型生成了一个高概率短语：

```text
Still hear you…
```

接下来：

- 这个短语被拼回输入
- 模型再次预测下一个 token
- 由于上下文几乎没变，这个短语仍然是“最优解”，在temperature = 0的情况下永远输出这个token

最终形成**确定性的无限重复**。
