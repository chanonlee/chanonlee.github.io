通用交易模型

最近在整理交易及相关上下游的业务模型，在这里试着写一下心得。

我们先把所有后端服务做一个粗糙的分类

  1. 计算型模块。如批处理计算，如配送路径优化，如 bom 拆解
  2. 状态机模块。如交易、支付、履约。
  3. 工作流模块。现在的业务 AI Agent 其实就应用了工作流，里面的核心是图结构。
  4. 基础数据模块。维护基础数据，如买家卖家信息，商品信息，公司信息。

交易是个典型的状态机模块，核心是领域单据的状态机，所有功能由状态机驱动。

交易的演进

交易就像是一个市场，有交易的对手双方，一般是一方交钱，一方交货，交易提供一个凭证记录，说明一方付了多少钱，一方交付了什么商品。这个时候的交易记录是没有状态的，成交就是成交，不成交就不成交。

在现代物流体系下，有可能一方已经交钱了，另一方要三天后货物才能交到对方手里，这时候就需要有个记录跟踪，看看货物是否真的到达用户手上了。这是履约的雏形。这时候交易单已经引入了状态机，支付可能是立即成功的，之后还得记录履约状态，是履约中还是履约成功了。

对于交易市场来说，买家非常多，卖家也非常多，这是一个 N 对 N 的交易模型。每一笔记录都需要记录买卖双方。交易所调用的支付体系来说，还需要处理不同买家、卖家使用的账户之间的差别。

领域建模

1. 主数据

我们将这个提供商品的一方称为卖家。

对于交易来说，一定有一个系统，提供买家信息，也有系统提供卖家信息，还有一个系统提供卖家所有可卖商品，形成如下图样。

买家主数据 -> 交易
卖家主数据 -> 交易
商品/服务主数据 -> 交易

2. 交易单

每一笔交易都有一个交易单，交易单上记录了这笔交易的情况，比如

  • 未支付
  • 已支付未履约
  • 已完成

另外，为了防止卖家无法履约，交易单一般都有自动关单时间，避免用户支付了一个半年前的交易单，此时卖家已经无法履约了，用户权益受损。

3. 交易子单

卖家提供多种服务，有商品，有咨询服务，也有代缴费服务，这是交易的不同业务入口。在这个业务入口下，每一种交易的支付和履约流程都不太一样，所以我们采用交易主单记录发生了一笔交易，用交易子单来记录具体的明细。比如商品有商品子单，咨询有咨询子单，代缴费有代缴费子单。

子单类型 履约特点 示例
商品子单 履约周期长，可能涉及物流的不同状态 用户购买手机，需等待快递配送
咨询子单 履约即时或周期短，状态单一 用户购买在线课程咨询
代缴费子单 履约同步下游系统，周期最短 用户代缴水电费，系统即时处理

说明：交易主单记录整笔交易，子单记录具体明细及履约情况，避免不同业务类型混杂导致逻辑混乱。

4. 支付情况

交易中，一般是先付款，再履约。而支付背后的复杂逻辑，银行、银行卡、时间节奏的相关信息由另一个支付模块隔离处理。交易一般只冗余存储是否完成支付。有些时候视系统查询需求，再看是否需要冗余付款人信息、付款时间信息等。

示例：用户下单购买商品时，交易模块只记录“已支付”，实际支付验证由支付模块完成。

5. 履约情况

支付流程相对统一，但不同交易子单的履约情况差异较大。

  1. 商品子单可能有多个履约方
    例如在电商系统中，一个商品订单可能由多个仓库或供应商发货。
    每个履约方的物流状态，如发货时间、运输进度、预计到达时间，都需要在子单中单独记录，确保查询和报表准确反映每个商品的履约情况。
  2. 服务子单通常只有一个履约方
    例如在线咨询、代缴费等服务订单，一般只涉及单一履约方，履约信息较为简单。

履约数据可根据系统查询需求冗余存储在交易子单上，便于快速查询和统计。
需要注意，这些冗余数据仅供参考，不保证百分百准确或实时。

6. 反向交易（退货/退款）

既然市场有交易，就得考虑反向交易，也就是用户退货退款的情况。这种情况下，为了保证退款的准确性，系统必须先校验正向交易是否存在、是否已完成，再根据当时交易的价格和数量进行退款。

换个思路想，这里也有买家卖家主数据，也有商品数据，只不过商品数据都由正向交易单上获取。反向交易就可以做成一个类似正向交易的系统了，也有主数据，也有交易主单和子单，也有支付（退款）和履约（退货）情况。一切按正向交易流程处理即可。

特别提醒，如果是一个数据量大且读多写少的系统，正向单据有时需要冗余反向交易情况。需特别记录，此时正单上的反向交易只是冗余，而非事实记录。
