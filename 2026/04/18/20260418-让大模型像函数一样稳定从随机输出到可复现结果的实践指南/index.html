在实际使用大模型时，我们有时候会遇到，问大模型同样的问题，大模型的回答每次都不一样，即使清除了上下文、使用了同样的模型、使用相同的prompt，回答也次次不同。这在工程场景中是一个很大的问题：难以调试、难以复现、难以稳定集成。

而这一切，往往由一个关键参数决定：temperature

1. 核心参数：temperature

llm模型每一步都会从概率分布中选择“最可能的下一个词”，而temperature 控制的是回答生成过程中的随机性。

  • temperature = 0 → 每一步都选择概率最高的 token
  • temperature > 0 → 在概率分布中进行随机采样

这里还有个很反直觉的事，llm并不是输入一个prompt，再输出一段话。它会把prompt和它已经输出的回答作为输入，再输出下一个字。

模型在根据自己已经说过的话继续往下说。这也是后面“死循环”问题的根本原因。

2. 候选词筛选——top_k和top_p

除了给token选择增加随机性，有时候人们也会给token设置筛选范围，避免随机到过分冷门的词，影响输出效果。这个筛选方案就是top_k和top_p。

top_k：只保留概率最高的 K 个 token，其他全部丢掉，然后在这 K 个里随机选。
top_p：按概率从高到低累加，直到总概率 ≥ p，保留这些 token。

3. 三个参数是如何配合的？

一句话总结：top_p / top_k 决定“选谁”，temperature 决定“怎么选”。

5. cursor等编程Agent里可以设置temperature参数吗？

不能直接设置。

在工程场景中，如果需要进行编程、数学计算等对结果一致性要求较高的任务，通常会将 temperature 设置为 0，以保证模型输出的稳定性和可复现性；而在生成图片描述、文本创作等更偏创意的任务中，则会适当提高 temperature，以获得更丰富、多样的结果。

大多数编程类 Agent（如 Cursor）为了避免代码生成过程中的不稳定性，通常会在内部预设合适的参数组合，并不对用户开放这些配置。可以理解为： 这些工具已经在底层帮你调好了参数。

6. temperature = 0带来的问题——死循环

稳定并不总是好事。

当 temperature = 0 时，模型每一步都会选择“当前最优解”，再将已输出的结果作为输入，生成下一个词。当重复一定次数，上下文稳定不再变化，模型就会持续输出重复词语，陷入死循环。
