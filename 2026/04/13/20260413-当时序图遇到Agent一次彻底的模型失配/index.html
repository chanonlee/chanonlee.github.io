当时序图遇到 Agent：一次彻底的模型失配

在尝试用传统软件工程工具去描述 Agent（特别是 ReAct 模式）时，我产生过一种很明显的违和感，图是能画出来，但总觉得哪里不对。

一开始以为是表达方式的问题，后来发现不是——是模型本身不兼容。

这篇文章就试图把这个“违和感”讲清楚。

一、时序图的本质：调用链 + 栈模型

二、Agent的本质：循环 + 状态机

当我们把这个模型套到 Agent 上，就开始不对劲了。ReAct中的循环最简单明确，我们以ReAct模式为例。

ReAct 的典型流程是：

  • 思考（Think）
  • 行动（Act）
  • 观察（Observe）
  • 再思考（Think）
  • 再行动……

结构如下：

这和调用栈最大的不同是，Agent 不关心“回到哪里”，只关心“下一步去哪”。更准确的类比是，Agent = 状态机（State Machine）。

特点：

  • 当前状态：由上下文（memory / messages）决定
  • 转移规则：由模型 + 决策函数决定
  • 没有“调用栈”，只有“状态演进”

四、更深一层：两种“程序”的哲学差异

如果说传统程序是在解决“如何把一条路径执行完”，那么 Agent 程序解决的，是“在图中下一步该走向哪里”。
前者是路径控制，后者是状态决策。
