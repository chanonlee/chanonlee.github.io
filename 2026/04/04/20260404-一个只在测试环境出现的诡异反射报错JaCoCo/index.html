最近在发布一个版本时发现：测试环境报错，生产环境正常；把测试环境回滚到几个月前的版本，问题依旧。排查到最后可以确定：不是业务代码回归，而是 JVM 挂载 JaCoCo 后向类里插入了额外字段，破坏了「按字段名拼 getter 再反射调用」的假设。本文记录完整排查过程，也方便以后按类似思路拆问题。

一、简化后的复现代码

下面这段逻辑会遍历类的所有声明字段，按 JavaBean 习惯拼出 getXxx，再用 invoke 调用——很多框架或自研工具里都有类似写法。

业务代码里没有显式引入任何与 JaCoCo 相关的包；前面几次回滚也证明，并不是某次代码改动「新引入」的问题。报错栈里若出现与 $jacocoData 相关的线索，排查重点自然会落到它上面。

二、从现象出发：$jacocoData 是什么？

先回忆一下 JaCoCo 是做什么的：用来做代码覆盖率统计。这个异常显然和它的工作方式有关。

在还不清楚机制时，可以先问一嘴 AI：这类报错通常怎么回事。大意是：在挂载了 JaCoCo 的 JVM 里，运行中的类会多出一个字段（例如 $jacocoData）；代码用字符串拼出 get$jacocoData 之类的名字去取方法，但并不存在对应的 getter，于是反射调用失败。

结合 JVM 的执行顺序，可以形成这样一个自洽的猜测：

  1. 业务里的 DTO 上，getter/setter 往往由 Lombok 在编译期生成进 .class。
  2. JaCoCo 挂在 JVM 上，在类加载等阶段做字节码插桩，时机比编译期更晚。
  3. 插桩会多出运行期才存在的字段，但不会再给你补一套 Lombok 式的 getter。
  4. 运行时若用反射遍历所有字段并拼 getter，就会把 $jacocoData 也算进去；去调 get$jacocoData 时就会发现无此方法，从而报错。

下面用一个小实验验证「挂载了 JaCoCo 的 JVM」是否足以复现。

三、第一轮实验：用「覆盖率运行」复现

要模拟「带 JaCoCo 的进程」，在 IntelliJ IDEA 里对 main 使用 Run with Coverage（使用覆盖率运行） 即可。执行参数里应能看到 **-javaagent**，且 agent 与 JaCoCo 相关。

仍用上一节的示例，对 main 做覆盖率运行：此时就会按预期出现与「为 $jacocoData 拼 getter」相关的失败。现象与测试环境一致，说明方向对。

四、JaCoCo 到底是怎么生效的？

问题大致定位并能复现之后，还想把图景补全：JaCoCo 是怎样挂到 JVM 上的？

我翻了 JaCoCo 官方文档，没有抠 API，直接看工作机制。文档里有一句关键描述（大意如下，原文为英文，此处为意译）：

JaCoCo 使用类文件插桩记录执行覆盖率。类文件通过所谓的 Java 代理（Java Agent） 做动态插桩。这样可以在类加载阶段在内存里预处理类文件，而不必绑死在某个应用框架上。

这里最关键的词就是：Java Agent。文档若不再展开，信息可以就停在这里；接下来顺着 Java Agent 往下追即可。

五、顺着 Java Agent 理解 $jacocoData

可以把 JVM 看成可扩展的运行平台：在业务 main 真正跑起来之前，允许先执行一段挂载逻辑。常见做法之一，就是在启动参数里加 **-javaagent:...**。

JVM 启动时若指定了 -javaagent，会先执行该 agent 包里的 premain 等入口；开发者可以在这个阶段对后续将要加载的类做字节码增强、插桩、监控等。

JaCoCo 正是利用这套机制：

  • 在类加载阶段对字节码插桩；
  • 插入用于记录覆盖率信息的字段与相关逻辑；
  • 运行结束后汇总成覆盖率报告。

这样一来，前面的 $jacocoData 就完全说得通了：它不是源码里写的字段，而是 JaCoCo 在运行时插进类里的字段。这类 Java Agent 在日常里也不少见，例如 JProfiler 等工具也是类似思路。
